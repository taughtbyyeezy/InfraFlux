# InfraFlux Project Context

This file serves as a permanent context anchor for the InfraFlux project, documenting its purpose, architecture, and technical foundations.

## Purpose
InfraFlux is a community-driven infrastructure reporting platform. It allows users to:
- **Report Issues**: Log potholes, water logging, and illegal garbage dumps.
- **Visualize Data**: View reports on an interactive map with real-time updates.
- **Validate Reports**: Use a voting system (upvotes/downvotes) to verify issue legitimacy.
- **Detailed Insights**: View evidence, severity, and reporting history for each issue.
- **Admin Moderation**: Secure dashboard for admin approval and removal of issues.

## Architecture
The system follows a modern decoupled architecture optimized for geospatial performance.

- **Frontend (SPA)**: A React-based application using Vite for fast builds and Leaflet for high-performance map rendering.
- **API Layer**: An Express.js server providing a RESTful interface, secured with Helmet and rate-limiting.
- **Data Persistence**: A PostgreSQL database utilizing the PostGIS extension for native spatial indexing and geometric operations.
- **Data Flow**: 
    1. **Frontend**: Captures location and metadata -> Sends JSON payload.
    2. **Backend**: Validates payload with Zod -> Executes transaction-safe SQL.
    3. **Database**: Updates tables and spatial indices -> Triggers auto-logic (approvals/delisting).

## Database Architecture
The database uses **PostgreSQL** with the **PostGIS** extension for spatial logic.

### Schema Overview
- **`issues`**: Stores the primary report data.
    - `geom`: `GEOMETRY(Point, 4326)` - The indexed spatial location.
    - `approved`: Boolean - Controlled by logic (20+ votes) or admin.
    - `votes_true` / `votes_false` / `resolve_votes`: Operational counters.
- **`issue_updates`**: State history (resolved, in_progress, etc.).
- **`issue_votes`**: Tracks unique votes via `voter_id` to prevent duplicates.
- **`media`**: Links visual proof (image URLs) to status updates.

### Core Queries
- **Spatial Retrieval**:
  ```sql
  SELECT ST_Y(geom) as lat, ST_X(geom) as lng FROM issues;
  ```
- **Latest Status Join**: Uses a `LATERAL JOIN` to fetch the most recent update for every issue during map rendering.
- **Auto-Logic (TSQL)**:
  - Approve: `UPDATE issues SET approved = TRUE WHERE votes_true >= 20`
  - Delist: `INSERT INTO issue_updates (status) VALUES ('resolved') WHERE votes_false >= 5`

## Technical Stack
- **Languages**: TypeScript (Strict), CSS (Vanilla), SQL.
- **Frameworks**: React 18, Express 4.
- **Geospatial**: Leaflet 1.9, React-Leaflet 4.2, PostGIS.
- **Validation/Utility**: Zod, date-fns, Lucide React.
- **Infrastructure**: Vercel (Frontend), Node.js (Runtime), PostgreSQL (Database).

## Constraints & Standards
- **UI/UX**: Strictly follows a "Glassmorphism" design system (Apple-inspired).
- **Responsive**: Mobile-first approach with sticky navigation and gesture-aware components.
- **Security**: Admin actions require a secret-header; Geolocation requires HTTPS for production usage.
- **Animations**: Performance-optimized using GPU-accelerated transforms and `will-change` hints.
